# CLAUDE.md

Guidance for Claude Code when working in this repository.

## What this repo is

`external_data_sources` is a multi-source monorepo of ingestion pipelines for
external (non-Avanti) public datasets, loaded into BigQuery for analysis.
One folder per upstream source. Each is self-contained: its own scripts,
its own raw + clean data, its own README and `CLAUDE.md`.

These pipelines are **not production services**. They run locally, on demand,
when an upstream publishes a new release. No orchestrator, no schedule.

Sources vary in shape вЂ” some need heavy local parsing (PLFS), some are

## When working on a specific source

Read that source's `<source>/CLAUDE.md` first вЂ” it has the codebase-level
orientation specific to that pipeline. The top-level CLAUDE.md (this file)
only covers cross-cutting conventions.

## Conventions across sources

- **One shared BigQuery dataset, source-prefixed tables.** All sources write
  into `avantifellows.external_data_sources.*`, with tables named
  `<source>_<kind>_<name>` вЂ” e.g. `plfs_fact_persons`, `nirf_fact_rankings`,
  `udise_dim_school` (when those exist). The dataset is in `asia-south1`.
- **GCS as the raw staging layer (when applicable).** For sources whose
  upstream is already analyst-clean (parquet/CSV/NDJSON), the raw files
  live at `gs://avantifellows-external-data/<source>/`, and the loader
  reads from GCS via `client.load_table_from_uri`. For sources that need
  real local parsing (PLFS), raw + clean stay on local disk and the
  loader uploads from a pandas DataFrame. Both patterns are first-class.
- **One-shot loader.** Each source has `scripts/load_bq.py` (or equivalent)
  that writes BQ tables. Idempotent. No DAG framework, no schedule.
- **Raw data gitignored; reference docs committed.** Manuals, layouts,
  codebook PDFs are small + public в†’ in git. Source data files are large
  or licensed в†’ gitignored. Each source's `.gitignore` enforces this.
- **Codemaps as CSV.** Small lookup tables (sex, sector, classification
  hierarchies) live in `<source>/codemaps/` as CSVs, regardless of whether
  they were typed by hand or generated by a script. CSV is the common
  interface downstream loaders consume.
- **Schemas as YAML.** One file per output BQ table in
  `<source>/schemas/`, named to match the table (e.g.
  `schemas/nirf_fact_rankings.yaml`). Documentation today; can be wired
  in as explicit BQ schemas later if type control is needed.
- **Python venv per source**, not at the repo root. Each source's
  README documents its setup. Different sources may use different Python
  versions or dependency managers.

## Adding a new source

1. Pick the closer reference: `plfs/` if upstream needs heavy local parsing,
   `nirf/` if upstream is already analyst-clean parquet/CSV. Copy that
   skeleton.
2. Strip out source-specific scripts, schemas, codemaps, raw docs.
3. Rewrite `<source>/README.md` and `<source>/CLAUDE.md` for the new source.
4. Add the source to the top-level `README.md` Sources table.
5. Implement `scripts/load_bq.py` that writes to
   `avantifellows.external_data_sources.<source>_*`.
6. Pick a GCS prefix under `gs://avantifellows-external-data/<source>/` if
   the source uses GCS staging.

## Don'ts

- **Don't put shared utility code at the root.** Sources vary too much.
  If the same 30 lines of BQ-load boilerplate end up in 3 sources, that's
  fine вЂ” premature factoring is worse than mild duplication here.
- **Don't split across BQ datasets.** Everything lives in
  `external_data_sources`; source separation is by table prefix, not by
  dataset. (Two sources can each have a `*_dim_geo` table вЂ” that's fine,
  they're just `plfs_dim_geo` and `udise_dim_geo`.)
- **Don't load raw data into git.** Each source's `.gitignore` is
  comprehensive; check it before adding under `raw/`.

## Reference implementations

Two patterns, two examples:

- [`plfs/`](plfs/) вЂ” heavy transform. Custom-format raw в†’ local parsing в†’
  pandas DataFrame в†’ BQ. Use when upstream isn't analyst-ready.
- [`nirf/`](nirf/) вЂ” light pass-through. Already-clean parquet в†’ GCS в†’
  BQ via `load_table_from_uri`. Use when upstream is analyst-ready.
- [`jnv/`](jnv/) вЂ” heavy transform with codemap-driven config. Raw Excel
  files в†’ schema-normalised CSV в†’ parquet в†’ GCS в†’ BQ. Introduces the
  codemap pattern: all year-specific column mappings live outside the
  engine script, making it zero-config for new years. Covers JEE Mains,
  JEE Advanced, NEET, JNVST selection test, EI Asset Test, and CBSE 10th/12th
  board results вЂ” six output tables: `jnv_fact_jee_results`,
  `jnv_fact_neet_results`, `jnv_fact_selection_test_results`,
  `jnv_fact_ei_asset_test_results`, `jnv_fact_board_results_10th`,
  `jnv_fact_board_results_12th`.

See `<source>/CLAUDE.md` for the full orientation of either.
