# external_data_sources

Ingestion pipelines for external (non-Avanti) public datasets that we load into
BigQuery for analysis. One folder per upstream source.

## Sources

| Source | Folder | Status | Description |
|---|---|---|---|
| PLFS (India) | [`plfs/`](plfs/) | Active | Periodic Labour Force Survey — MoSPI unit-level microdata across 11 releases (2018-19 → CY2025). 6-table BQ dataset (`plfs`). |

Future candidates: UDISE (school stats), NFHS (health), ASER (learning levels),
Census 2011 (and 2021 when released). Add a new folder per source.

## Conventions for adding a new source

Each `<source>/` folder is self-contained: its own README, its own `CLAUDE.md`,
its own scripts, its own raw + clean data. No shared code across sources —
they vary too much for that to be useful at this scale.

Recommended skeleton (copy from `plfs/`):

```
<source>/
├── README.md          # what the source is, how to fetch it, how to load
├── CLAUDE.md          # Claude orientation for this source
├── scripts/
│   └── load_bq.py     # one-shot ETL → BigQuery
├── schemas/           # YAML docs for each output table
├── codemaps/          # small lookup tables, committed
├── raw/               # source files
│   ├── docs/          # layout docs, manuals (committed if small + public)
│   └── data*/         # raw downloads (gitignored — usually licensed)
├── clean/             # parsed intermediate artifacts (mostly gitignored)
├── analyses/          # exploratory scripts (optional)
└── .gitignore         # source-specific ignores
```

### Defaults worth keeping consistent

- **One BQ dataset per source** (e.g. `plfs`, `udise`). Don't mix.
- **One-shot loader** (`scripts/load_bq.py`) — no orchestrator, no scheduler.
  These pipelines run on demand when upstream publishes a new release.
- **Raw data gitignored, reference docs committed.** Public manuals/layouts
  belong in git; gated/licensed data files don't.
- **Codemaps committed as CSV.** Even if generated by a script, the CSV is
  the consumed artifact and the common interface across sources.
- **Schemas as YAML** in `<source>/schemas/`, one file per output table.
- **Python via `uv` or `venv` per source.** Each source has its own
  dependencies; don't share a root `.venv`.

### Top-level only

The repo root holds:
- this README,
- `CLAUDE.md` (Claude orientation across the monorepo),
- a thin `.gitignore` for cross-cutting noise (`.DS_Store`, `__pycache__/`,
  `.venv/`, `.claude/`).

Everything else lives under a source folder.

## Reference implementation

[`plfs/`](plfs/) is the reference. New sources should mirror its structure.
See [`plfs/README.md`](plfs/README.md) for the worked example and
[`plfs/CLAUDE.md`](plfs/CLAUDE.md) for the codebase orientation.
